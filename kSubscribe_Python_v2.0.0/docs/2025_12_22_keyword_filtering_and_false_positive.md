## 1. 문제 정의

### 1.1 문제 상황

**현상**: 
- 한국전력 관련 기사에서 "한국전력", "한전" 같은 기관명 표현이 많이 나옴
- 이로 인해 `predefine_keyword`에서 "전력"이 잘못 선택되는 경우가 발생

**구체적 사례**:
- ✅ **올바른 선택**: "원전 가동 중단으로 전력 공급에 차질" → "전력" 선택 (적절)
- ❌ **잘못된 선택**: "한전 식당 메뉴 부실 논란" → "전력" 선택 (부적절, 기사 주제와 무관)

### 1.2 문제의 본질

**문제 유형**: **False Positive (거짓 양성) - 기관명 기반 오선택**

**근본 원인**:
1. **기관명과 주제 키워드의 혼동**
   - LLM이 "한국전력"이라는 기관명을 보고 "전력"이라는 주제 키워드와 자동 연결
   - 기사 내용의 실제 주제(식당 메뉴, 급식 등)는 무시됨

2. **문맥 무시**
   - 기관명이 언급되었다는 이유만으로 관련 키워드가 선택됨
   - 기사가 다루는 실제 주제와 키워드의 의미적 연결이 약함

3. **프롬프트의 모호성**
   - 현재 프롬프트가 "의미적으로 연결되는 항목"이라고만 지시
   - 기관명과 주제 키워드를 구분하는 명확한 기준이 없음

### 1.3 문제 정의

**정식 명칭**: **"기관명 기반 주제 키워드 오선택 문제" (Organization Name-based Topic Keyword False Positive)**

**핵심 이슈**:
- 기관명이 `predefine_keyword`의 일부를 포함할 때, 기사 주제와 무관하게 해당 키워드가 선택됨
- 예: "한국**전력**" → "**전력**" 키워드 오선택
- 예: "한국**에너지**공단" → "**에너지**" 키워드 오선택

## 2. 현재 시스템 동작 분석

### 2.1 전체 프로세스 개요

PredefineKeyword 필터링은 **2단계**로 이루어집니다:

1. **1단계: LLM 기반 사전 필터링** (`question_verify` 프롬프트)
   - 기사에서 `ai_keyword` 추출
   - `ai_keyword`와 `predefine_keyword` 비교하여 관련성 검증
   - 관련된 `predefine_keyword`만 `reason` 필드에 저장 (최대 10개)

2. **2단계: 유사도 기반 최종 선택** (`SimularityChecker`)
   - `short_summary`와 `reason` 키워드들 간 유사도 계산
   - 상위 3개 선택 (threshold 없음)

### 2.2 1단계: LLM 기반 사전 필터링

#### 현재 프롬프트의 문제점

**`question_verify` 프롬프트 (현재)**:
```
1. 주제 파악: 기사 내용을 분석하여 핵심 이슈나 주제를 나타내는 '키워드'를 먼저 추출해라.
2. 매칭 확인: 추출한 '키워드'와 [DB 키워드 리스트]를 비교하여, 의미적으로 연결되는 항목이 있는지 확인해라.
```

**문제점**:
- "의미적으로 연결"이라는 표현이 모호함
- 기관명과 주제 키워드를 구분하지 않음
- 기사 내용의 실제 주제를 강조하지 않음

#### 실제 발생 시나리오

**시나리오 1: 원전 관련 기사 (올바른 선택)**
```
기사: "한국전력이 원전 가동 중단으로 전력 공급에 차질이 생겼다."
ai_keyword: ["한국전력", "원전", "가동중단", "전력공급"]
매칭: "전력" ✅ (올바름 - 기사 주제가 전력 공급)
```

**시나리오 2: 식당 메뉴 관련 기사 (잘못된 선택)**
```
기사: "한국전력 직원 식당에서 메뉴 부실 논란이 발생했다."
ai_keyword: ["한국전력", "식당", "메뉴", "부실"]
매칭: "전력" ❌ (잘못됨 - 기사 주제는 식당/급식, 전력과 무관)
```

### 2.3 2단계: 유사도 기반 최종 선택

#### 유사도 계산 메서드

**파일**: `simularity_check.py`

```python
def best_keyword_of_summary(self, title: str, predefineKeyword: List[str]): 
    # 1. SentenceTransformer 모델 사용
    # 모델: 'sentence-transformers/paraphrase-multilingual-MiniLM-L12-v2'
    
    # 2. short_summary를 벡터로 변환 (768차원)
    title_embedding = self.model.encode(title, convert_to_tensor=True)
    
    # 3. keywords_verify 리스트를 벡터로 변환 (N×768차원)
    keyword_embeddings = self.model.encode(predefineKeyword, convert_to_tensor=True)
    
    # 4. 코사인 유사도 계산
    cosine_scores = util.cos_sim(title_embedding, keyword_embeddings)
    
    # 5. 상위 K개 선택
    top3_indices = torch.topk(
        cosine_scores[0], 
        k=min(3, len(predefineKeyword))
    ).indices
    
    # 6. 딕셔너리 생성 {키워드: 유사도점수}
    dic = {}
    for index in top3_indices:
        keyword = predefineKeyword[index]
        dic[keyword] = cosine_scores[0][index].item()
    
    return dic
```

#### 핵심 설정값

| 항목 | 값 | 비고 |
|------|-----|------|
| **전체 predefine_keyword 수** | 29개 | 모든 기관에 동일 |
| **1단계 최대 선택** | 10개 | `reason` 필드 최대 개수 |
| **2단계 최대 선택** | 3개 | `best_keyword_of_summary` |
| **유사도 threshold** | **없음** | 점수와 관계없이 상위 3개 선택 |
| **유사도 범위** | 0.0 ~ 1.0 | 코사인 유사도 |
| **임베딩 모델** | paraphrase-multilingual-MiniLM-L12-v2 | 768차원 |

## 3. 해결 방안 연구

### 3.1 방안 1: 기관명 제외 및 주제 중심 검증

**핵심 아이디어**: 
- 기관명 자체는 키워드 매칭에서 제외
- 기사가 다루는 **실제 주제/이슈**만 키워드와 매칭

**장점**:
- 기관명 기반 오선택 방지
- 주제 중심의 정확한 매칭

**단점**:
- 기관명이 실제 주제와 관련된 경우도 있음 (예: "한국전력의 전력 정책")

### 3.2 방안 2: 문맥 기반 엄격한 의미 연결 검증

**핵심 아이디어**:
- 키워드가 기사 **내용의 핵심 주제**와 직접적으로 관련되어야 함
- 단순히 기관명에 포함되어 있다고 선택하지 않음

**장점**:
- 문맥을 고려한 정확한 매칭
- False Positive 감소

**단점**:
- 프롬프트가 복잡해질 수 있음

### 3.3 방안 3: 2단계 검증 (기관명 필터링 + 주제 검증)

**핵심 아이디어**:
1. 1단계: 기관명과 키워드의 표면적 매칭 확인
2. 2단계: 기사 내용의 실제 주제와 키워드의 의미적 연결 검증

**장점**:
- 단계별로 명확한 검증
- 높은 정확도

**단점**:
- 프롬프트가 길어짐
- 처리 시간 증가 가능

### 3.4 방안 4: 부정 예시 제공 (Few-shot Learning)

**핵심 아이디어**:
- 프롬프트에 올바른 예시와 잘못된 예시를 함께 제공
- LLM이 패턴을 학습하도록 유도

**장점**:
- 명확한 기준 제시
- 학습 효과

**단점**:
- 프롬프트 길이 증가
- 예시 선택이 중요

### 3.5 권장 해결 방안

**추천**: **방안 2 + 방안 4 조합** (문맥 기반 검증 + 부정 예시)

**이유**:
1. 기관명 제외만으로는 부족 (기관명이 실제 주제와 관련될 수 있음)
2. 문맥 기반 검증이 가장 정확함
3. 부정 예시로 명확한 기준 제시

## 4. Verify 프롬프트 수정 방안

### 4.1 방안 A: 문맥 중심 엄격한 검증 (권장)

#### 핵심 변경사항
1. **기관명과 주제 키워드 구분 명시**
2. **기사 내용의 실제 주제 강조**
3. **부정 예시 제공**

#### 수정된 프롬프트 구조

```python
question_verify_v2 = f"""
Reasoning: high
# Valid channels: analysis, final

너는 뉴스 기사와 사전 정의된 키워드 간의 연관성을 검증하는 '데이터 매칭 전문가'다.
아래 [기사]의 **실제 주제/이슈**를 파악하고, 제공된 [DB 키워드 리스트]와 매칭되는 항목이 있는지 검증해라.

### [중요: 기관명과 주제 키워드 구분]
- **기관명**: "한국전력", "한전", "한국에너지공단" 등은 단순히 기관을 지칭하는 이름이다.
- **주제 키워드**: 기사가 실제로 다루는 주제나 이슈를 나타내는 단어다.
- **주의**: 기관명에 키워드가 포함되어 있다고 해서 자동으로 선택하지 마라.
  예: "한국전력"이라는 기관명이 나왔다고 해서 무조건 "전력"을 선택하지 마라.

### [검증 가이드라인]
1. **실제 주제 파악**:
   - 기사가 실제로 다루는 핵심 주제나 이슈를 파악해라.
   - 기관명은 제외하고, 기사 내용의 주제만 고려해라.
   - 예: "한국전력 식당 메뉴 부실" → 주제는 "식당/급식", "전력"과 무관

2. **의미적 연결 검증**:
   - 추출한 주제 키워드와 [DB 키워드 리스트]를 비교하여, **기사 내용의 주제와 직접적으로 관련된** 항목만 선택해라.
   - 단순히 기관명에 포함되어 있다는 이유로 선택하지 마라.

3. **엄격한 선택 규칙**:
   - `reason` 필드에는 **반드시 [DB 키워드 리스트]에 포함된 단어만** 기재해야 한다.
   - 기사 본문에 있는 단어라도 [DB 키워드 리스트]에 없다면 `reason`에 넣지 마라.
   - **기관명에 키워드가 포함되어 있어도, 기사 주제와 무관하면 선택하지 마라.**

4. **결과 판정**:
   - 연관된 키워드가 1개 이상이면 `related`를 `true`로 설정해라.
   - 연관된 키워드가 전혀 없으면 `related`를 `false`로 설정해라.

### [부정 예시 - 선택하면 안 되는 경우]
- ❌ 기사: "한국전력 직원 식당에서 메뉴 부실 논란"
  - 잘못된 선택: "전력" (기사 주제는 식당/급식, 전력과 무관)
  - 올바른 판단: related = false 또는 reason = []

- ❌ 기사: "한국에너지공단 신입사원 채용 공고"
  - 잘못된 선택: "에너지" (기사 주제는 채용, 에너지와 무관)
  - 올바른 판단: related = false 또는 reason = []

### [긍정 예시 - 선택해야 하는 경우]
- ✅ 기사: "한국전력 원전 가동 중단으로 전력 공급 차질"
  - 올바른 선택: "전력", "에너지" (기사 주제가 전력 공급과 직접 관련)

- ✅ 기사: "한국에너지공단 신재생에너지 정책 발표"
  - 올바른 선택: "에너지", "신재생" (기사 주제가 에너지 정책과 직접 관련)

### [기사]
[contents]

### [DB 키워드 리스트]
[pred_keywords_from_db]

### [출력 형식]
반드시 아래 JSON 포맷으로만 응답해라. 주석, 설명은 절대 포함하지 마라.

{{
    "ai_keyword": ["기사에서 추출한 핵심 주제 키워드 리스트"],
    "db_keyword_list": ["입력받은 DB 키워드 리스트 그대로 반환"],
    "related": true 또는 false,
    "reason": ["DB 키워드 리스트 중 기사 주제와 직접적으로 연관된 단어들 (최대 10개)"]
}}
"""
```

#### 변경 포인트
1. ✅ **기관명과 주제 키워드 구분** 섹션 추가
2. ✅ **실제 주제 파악** 강조
3. ✅ **부정 예시** 2개 추가
4. ✅ **긍정 예시** 2개 추가
5. ✅ "기관명에 포함되어 있어도 주제와 무관하면 선택하지 마라" 명시

## 5. 기관명 제거 방식

### 5.1 핵심 아이디어

**전략**: 기관명을 제거하되, 기관의 행동/정책/이슈는 보존하여 키워드 매칭

**예시**:
- ❌ "한국전력 식당 메뉴 부실" → "식당 메뉴 부실" (기관명 제거, 주제만 남음)
- ✅ "한국전력의 전력 정책 발표" → "전력 정책 발표" (기관명 제거, 정책은 남음)

### 5.2 처리 예시

| 원본 기사 표현 | 처리 후 | 키워드 매칭 결과 |
|---------------|---------|----------------|
| "한국전력 식당 메뉴 부실" | "식당 메뉴 부실" | ❌ "전력" 선택 안 됨 (주제와 무관) |
| "한국전력의 전력 정책 발표" | "전력 정책 발표" | ✅ "전력" 선택 가능 (주제와 관련) |
| "한전 원전 가동 중단" | "원전 가동 중단" | ✅ "전력", "원자력" 선택 가능 |
| "한국에너지공단 신입사원 채용" | "신입사원 채용" | ❌ "에너지" 선택 안 됨 |
| "한국에너지공단 신재생에너지 정책" | "신재생에너지 정책" | ✅ "에너지", "신재생" 선택 가능 |

### 5.3 기관명 제거 구현

#### 방안 1: 기관 정보 활용 (권장)

**장점**: 이미 코드에서 기관 정보를 가져오고 있으므로 활용 가능

```python
question_verify_v3_org_aware = f"""
### [대상 기관 정보]
- 기관명: [organization]

### [핵심 원칙: 기관명 제거 + 행동/정책 보존]

1. **기관명 제거**:

   - 예: "한국에너지공단", "에기평" → 제거

2. **행동/정책/이슈 보존**:
   - 기관명을 제거하되, 기관의 행동, 정책, 이슈, 사건 등은 그대로 보존해라.
   - 보존해야 할 것: 정책, 사업, 이슈, 사건, 행동, 발표, 계획 등
   - 예: "한국전력의 전력 정책 발표" → "전력 정책 발표" (기관명 제거, 정책은 보존)
   - 예: "한전 원전 가동 중단" → "원전 가동 중단" (기관명 제거, 이슈는 보존)

3. **보존된 내용으로 매칭**:
   - 기관명을 제거한 후 남은 내용만으로 키워드 매칭을 수행해라.
```
